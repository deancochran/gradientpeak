## Research Report: The Optimal Open Standard for Performance Activity Data

**To:** Interested Parties
**From:** Research Division
**Date:** January 20, 2026
**Subject:** Analysis of Open Standards for Recording and Parsing Athletic Performance Data

### 1. Introduction

This report addresses the need for a robust, open-source, and publicly known standard for handling athletic activity data . The primary goal is to identify an alternative to using custom compressed JSON payloads, which necessitate bespoke server-side decompression and parsing logic . The ideal solution must be efficient, comprehensive, and ensure broad compatibility with major third-party fitness platforms, including Garmin, Wahoo, and Strava. The standard must support detailed performance analysis metrics such as Training Stress Score (TSS), Intensity Factor (IF), Functional Threshold Power (FTP), and VO2max, either directly or through the inclusion of underlying raw data. This report will also provide a practical implementation blueprint for the recommended standard [[1]](https://storage.cloud.google.com/essent-agentspace-test2/New%20folder/Arkansas%20Rate%20Manual%20Effective%2002-01-2023.pdf?authuser=0) .

This research examines the most prevalent data formats in the fitness technology industry: **Flexible and Interoperable Data Transfer (FIT)**, **Training Center XML (TCX)**, and **GPS Exchange Format (GPX)**. Each standard is evaluated based on its data richness, efficiency, industry adoption, openness, and the availability of development tools.

### 2. Candidate Data Standards

#### 2.1. GPX (GPS Exchange Format)

The GPS Exchange Format (GPX) is a light-weight, open standard designed for the interchange of GPS data. It is an XML-based schema that is human-readable and widely supported across almost all mapping and GPS-related applications.

*   **Structure:** GPX files are structured around three primary data types: waypoints (single points), routes (a sequence of points to follow), and tracks (a recorded path). A track is composed of track segments, which contain a series of time-stamped track points with latitude, longitude, and elevation.
*   **Data Richness:** The core GPX standard is primarily focused on location and time. While it can be extended to include other data like heart rate and cadence using custom XML extensions, this data is not part of the core specification. Support for these extensions is inconsistent across platforms, and the format is not designed to natively handle complex cycling dynamics or summary performance metrics like TSS or NP.
*   **Compatibility:** It is the universal standard for sharing routes and basic GPS tracks. All major platforms, including Strava, Garmin, and Wahoo, can import and export GPX files. However, they may not interpret all extended data fields.

#### 2.2. TCX (Training Center XML)

Developed by Garmin, Training Center XML (TCX) was created to expand upon the limitations of GPX for fitness-specific applications. It is also an XML-based format with a publicly available schema.

*   **Structure:** TCX is more activity-centric than GPX. It organizes data into "Activities," which contain "Laps" and "Tracks." Each track point can natively include not just GPS data, but also heart rate, distance, and cadence. Some versions of the schema also include extensions for power data.
*   **Data Richness:** TCX offers a significant improvement over GPX for performance analysis, with first-class support for heart rate and cadence. It can store the necessary raw data to calculate metrics like TSS and IF. However, it does not have standardized, universally recognized fields for these summary metrics themselves; they are typically calculated by the platform upon import.
*   **Efficiency:** As an XML-based format, TCX files are human-readable but are significantly larger than binary formats. For a given activity, a TCX file can be many times the size of a FIT file, making it less efficient for storage and transmission.

#### 2.3. FIT (Flexible and Interoperable Data Transfer)

The FIT protocol, also originating from Garmin, is a binary file format designed to be the most comprehensive and efficient standard for sport, fitness, and health data. It has become the de facto standard for modern fitness devices.

*   **Structure:** FIT files are composed of a series of "messages," which are compact binary data records [[2]](https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQH_w0GHEnval0MvEhp7V5md93ppHjFH6yP_0-Fku1tIRDB-dYz7izVIjCuIaln8M-L6vz8AsNBUvkeSSUrRB-i7o2dEBmzC65s3tIAIB5nsHOU39X3HQ4mw7IQ9fC_cJgxuJ9bjUvW3gWdSUJSfJuvKIg_KThe814yyFr3Bgg==) . Key message types include `file_id`, `device_info`, `record` (for per-second data points), `lap` (for lap summaries), and `session` (for the overall activity summary) [[3]](https://storage.cloud.google.com/essent-agentspace-tes/Essent-test-files/W2/Form_W-2%2C_2016.pdf?authuser=0)[[2]](https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQH_w0GHEnval0MvEhp7V5md93ppHjFH6yP_0-Fku1tIRDB-dYz7izVIjCuIaln8M-L6vz8AsNBUvkeSSUrRB-i7o2dEBmzC65s3tIAIB5nsHOU39X3HQ4mw7IQ9fC_cJgxuJ9bjUvW3gWdSUJSfJuvKIg_KThe814yyFr3Bgg==) . This message-based structure is highly extensible.
*   **Data Richness:** This is the most significant advantage of the FIT format. It natively supports a vast array of data fields, including:
    *   **Core Metrics:** GPS, altitude, speed, distance, heart rate, cadence, and temperature.
    *   **Power Metrics:** A comprehensive set of power data, including average power, max power, and left/right balance.
    *   **Summary Metrics:** The FIT protocol explicitly defines fields within its `session` and `lap` messages for key performance indicators [[4]](https://storage.cloud.google.com/essent-agentspace-tes/Essent-test-files/David-files/URLA_2019_Borrower_v28.pdf?authuser=0) . These include `training_stress_score` (TSS), `intensity_factor` (IF), and `normalized_power` (NP) [[3]](https://storage.cloud.google.com/essent-agentspace-tes/Essent-test-files/W2/Form_W-2%2C_2016.pdf?authuser=0) . It also includes fields for user data like `threshold_power` (FTP).
    *   **Developer Data:** The format allows for custom "Developer Fields," enabling manufacturers and apps (like those from the Connect IQ ecosystem) to write their own specific data into the file without breaking the standard.
*   **Efficiency:** Being a binary format, FIT files are extremely compact. They are significantly smaller than their TCX or GPX counterparts for the same activity, making them ideal for on-device storage and efficient data transfer, which directly addresses the goal of replacing custom compressed JSON.

### 4. Openness and Developer Support

*   **GPX:** A true open standard.
*   **TCX:** Publicly available XML schema provided by Garmin. Can be parsed with any standard XML library.
*   **FIT:** The protocol is publicly documented by Garmin. A royalty-free **FIT SDK** is available, which includes source code for encoding and decoding in C++, Java, C#, and Python [[5]](https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQFgu13sywpDbzsUdY8XtWlZ6xfMNAnGpy1q0_UFTNzavLCvdS0CJP5ynaKCrLjN08dopozKQL-snJCbSgWp0tIVLZexqOEt35bzx8mES6iOw6jcDRZJruZ6SAoZD9KMjCR92Svoa69ad62gAtNfeRm61VDWk20lNks3kgKNISW-ydrbHVK1LZR2i0wEsjb_6jAGH5DVKmoPm-LyYEzjqR3HiTQbbyYFpVbJmw==) . #### 5.1. Client-Side: Fault-Tolerant Recording and Encoding (React Native)

A fault-tolerant strategy prevents data loss by persisting data to a temporary file during the recording session .

*   **Temporary File Persistence:** Upon starting a recording, create a single temporary file (e.g., `pending_activity.json`) in the application's cache directory using a library like `react-native-fs` [[9]](https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQFieSzOO7DBTHt-D0zMnK-dCkGeqOMEzPn7n3d7DzFU3mLw884HPgRPKSwqOz9BftJFpLjiG7x1QF8LXMFDJs646GoUbkIny1tIDQPQCq1Nb2PPVfhm4IbuGKkfYdF2l_1f2jTbN49sHzyxQG-7MRCR1BC4az_ACh4akxpc1w3oMmBzkQwhmk_VaF8B7_u-4cHx1Ld_xtqAK3pXpArLpA==) . *   **Crash Recovery:** Upon application startup, the system must check for the existence of this temporary file [[10]](https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQF9tNE9aECShqVkvJSuqn8XwrxcBrxs37GIMSNOQMchDQXqppEaXdyJ9uqEgBQjmo9JBq05sL2C5WGXeoNe-I4okMPmoftWIHGofVBSwjyw9DINoDaniZSgi47t1QyrdsHjf8y-MkDoyDAk)[[11]](https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQFB9LGJ7yUN47S8p-i4kIDoAx_bLnCCOjU2yTaTmjvJ366o5kAOIbFgVHxxiWl6qKY350mqyKVDxz1RRpcT1FHUF7KEXb__8E6bfafEI-OZN3VdELNLWAWmgkiahxz2OwVo4OKU3J2TG9kPIjOA2DDcblMQcptTn50EGA==) . ### 6. Recommendation

Based on the research, the **FIT (Flexible and Interoperable Data Transfer) protocol is the superior and recommended standard** for the specified requirements.

It is the only standard that directly addresses all of the project's needs:

1.  **Alternative to Custom Compressed JSON:** As a compact binary format, FIT provides the efficiency needed to replace a custom compressed payload without the overhead of a text-based format like TCX or GPX. The pre-signed URL architecture provides a scalable submission mechanism [[12]](https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQFPJiAEtZ0_VFPUvuD9_sLQ1Y4SwKULsxnFZucM2qocAmDajTxhzhYKJQ7hT9UluZ_81ukTrNJrc0Rj9IIEXqheX64MZoTVmtnrfOr-wUHprH0tKdHVEqMMCYaw)[[2]](https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQH_w0GHEnval0MvEhp7V5md93ppHjFH6yP_0-Fku1tIRDB-dYz7izVIjCuIaln8M-L6vz8AsNBUvkeSSUrRB-i7o2dEBmzC65s3tIAIB5nsHOU39X3HQ4mw7IQ9fC_cJgxuJ9bjUvW3gWdSUJSfJuvKIg_KThe814yyFr3Bgg==) .

### 7. Executive Summary

The objective was to find an open, public, and standardized data format for athletic activities that could replace a custom compressed JSON system, while ensuring compatibility with Garmin, Wahoo, and Strava and supporting key performance metrics (TSS, IF, FTP, etc.).

After evaluating GPX, TCX, and FIT formats, the **FIT protocol is the clear and definitive recommendation**.

The FIT format's binary structure provides the **efficiency and compactness** required, directly replacing the need for custom compression.
