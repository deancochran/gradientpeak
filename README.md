## üèóÔ∏è Architecture Overview

TurboFit is organized as a **Turborepo monorepo** with modular packages:

### üì¶ Core Package (`packages/core`)

The heart of TurboFit, shared across web, mobile, and backend apps. **Completely independent of database or ORM dependencies.**

**Responsibilities:**

* **Type Definitions & Schemas** ‚Äî Zod validation for profiles, activities, activities, and flexible JSON structures
* **Calculations & Analytics** ‚Äî Training zones, TSS, normalized power, compliance scoring, CTL/ATL/TSB
* **Business Logic** ‚Äî Activity plan validation, progression, and adaptive algorithms
* **Utilities** ‚Äî Time/duration helpers, unit conversions, constants
* **Platform Agnostic** ‚Äî Pure TypeScript with no database, ORM, or platform-specific dependencies

**Key Benefit:** Single source of truth ensures consistent calculations, type safety, and validation across all applications while remaining completely portable and testable in isolation.

---



### üì¶ TypeScript Config Package (`packages/typescript-config`)

A shared TypeScript configuration used across all apps and packages in the TurboFit monorepo.

**Responsibilities:**

* **Centralized TS Configuration** ‚Äî Base `tsconfig` defines compiler options, strict type checking, and module resolution
* **Standardized Paths & Aliases** ‚Äî Ensures consistent `@/*` imports across apps
* **Extensible per App** ‚Äî Apps can extend the base config for Next.js, Expo, or library-specific overrides
* **Version Control** ‚Äî Single source of truth for TypeScript settings to reduce discrepancies and errors

**Key Benefit:** All apps and packages share a **consistent TypeScript environment**, simplifying cross-package type safety and refactoring.

* **Custom Plugins** ‚Äî Turbo-specific rules, `eslint-plugin-only-warn`, and other shared rules enforced consistently
* **Ignored Paths** ‚Äî Standard exclusions for `dist`, `.next`, and build artifacts

**Key Benefit:** Provides **uniform code quality standards** across all apps and packages, making maintenance and onboarding simpler.

## üßë‚Äçüíª Profiles & Preferences

User-centric design anchors the system around **profiles** extended from `auth.users`. The core package provides enhanced profile interfaces and validation schemas for athlete-specific metrics:

| Field             | Description                                         |
| ----------------- | --------------------------------------------------- |
| `id`              | Primary key, UUID, FK ‚Üí `auth.users.id`             |
| `threshold_hr`    | Threshold heart rate (bpm); nullable for new users  |
| `ftp`             | Functional Threshold Power (watts); nullable        |
| `weight_kg`       | Athlete's weight for power-to-weight calculations   |
| `gender`          | Used in predictive models and analytics             |
| `dob`             | Date of birth; calculates age-based zones & targets |
| `username`        | Unique public-facing handle                         |
| `language`        | Preferred UI language/locale                        |
| `preferred_units` | Metric vs imperial                                  |
| `avatar_url`      | Optional profile picture/avatar                     |
| `bio`             | Optional short biography                            |

The core package handles **profile validation**, **training zone calculations**, and **unit conversions** for personalized experiences.

## üèÉ Activity Storage & Performance Analysis

### JSON-First Activity Architecture

TurboFit uses a **JSON-first approach** where all activity data is stored as the single source of truth:

* **Primary Storage** ‚Äî Complete activity data stored as JSON in Supabase Storage
* **Local Recording** ‚Äî Activities initially captured locally in SQLite as JSON
* **Cloud Sync** ‚Äî JSON objects uploaded to Supabase Storage when network available
* **Metadata Generation** ‚Äî Activity records created locally and synced after JSON storage
* **Stream Processing** ‚Äî Activity streams generated from JSON after successful upload

**Data Flow:**
1. **Record** ‚Üí Local SQLite stores complete activity as JSON
2. **Upload** ‚Üí JSON object uploaded to Supabase Storage (source of truth)
3. **Process** ‚Üí Activity metadata record generated locally and inserted
4. **Streams** ‚Üí Activity streams generated and inserted after JSON processing
5. **Analytics** ‚Äî Core package processes JSON for performance calculations

### Performance Analysis

Activity data flows through the **core package** for consistent analysis:

* **Performance Metrics** ‚Äî TSS, normalized power, intensity factors calculated from JSON
* **Training Load Analytics** ‚Äî CTL, ATL, TSB derived from activity metadata
* **Compliance Scoring** ‚Äî Activity matching algorithms using JSON activity structure
* **Zone Analysis** ‚Äî Training zone calculations using core package algorithms

### Activity Streams

Time-series data processed from JSON source through core package utilities:

* **Generated from JSON** ‚Äî All streams derived from primary JSON activity data
* **Standardized Metrics** ‚Äî Core package ensures consistent metric types and validation
* **Performance Curves** ‚Äî Power/HR curves calculated from JSON streams
* **Real-time Processing** ‚Äî Stream aggregation during activity recording
